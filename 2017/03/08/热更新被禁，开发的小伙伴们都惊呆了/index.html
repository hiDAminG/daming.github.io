热更新被禁，开发的小伙伴们都惊呆了

  • iOS的日常

使用 JSPatch

被苹果警告，热修复未来何去何从

JSPatch

的由来

JSPatch
诞生于2015年5月，最初是腾讯广研高级iOS开发 @bang 的个人项目。
它能够使用JavaScript调用Objective-C的原生接口，从而动态植入代码来替换旧代码，以实现修复线上bug。
JSPatch
在Github.com上开源后获得了3000多个star和500多fork，广受关注，目前已被应用在大量腾讯/阿里/百度的App中。

总的来说JSPatch
是一个可以提供方法水平热修复的工具，而且他是利用苹果提供JavaScriptCore的功能工作的。而且效率高还小巧，深受各大公司的喜爱。

那为什么这样一个广受喜爱的而且还是利用原生提供的功能实现的开发工具会受到苹果的警告呢？？

首先苹果本身是不允许热修复的，所以 JSPatch
的存在本身就带有一定政策风险。而这次苹果给出的警告解释是 JSPatch
的脚本存在被拦截替换的风险，所以苹果给出警告！

事情到这里本来就很清楚明了了。
苹果已经开始对 JSPatch
的广泛使用产生忧虑了。开始慢慢建议开发者不要再使用这种方式了。所以那天为此让开发者下架也不是没有可能！

事已至此 本身事情都很明朗了，但网上却涌出一大批高呼 React-native药丸的声音！
这又是怎么一回事？？

React-native们与热更新

React-native本身是可以只作为原生开发的框架使用的。但 React-native与js天生的亲密 使得 React-native的一大特色就是热更新敏捷开发。

热更新已经成为了 React-native的一大亮点，和 JSPatch
结合的也是风生水起。
大批公司也因此开发了大量的 类React-native应用，其中不乏大厂的顶梁柱应用。
React-native生态本来是欣欣向荣。
但现在苹果向热更新开到，向世人展示了一个信号，似乎预示着某种收紧的预兆。这种情况下本来就是优点和缺点都很突出 React-native收到的影响也是首当其冲。
估计很多厂子会在接下来的一段时间谨慎考虑 React-native的应用。生态区会有萎缩的风险。

另一方面，苹果向来我行我素。今天他想热更新开刀。明天保不齐就向混合开发开炮！也着实让开发者们有点存亡齿寒。

所以对 React-native的担忧也不全是空穴来风。

当然从另一个角度想想，苹果为什么早不开刀晚不开刀，偏偏现在开刀？

也可能是已经准备了另一套的工具，现在只是在为他开路而已~~
当然我不此猜测负责~~
